摘要

我们介绍 $\pi^{3}$，一种前馈神经网络，打破了对传统固定参考视图的依赖。之前的方法通常将重建锚定在指定的视点之上，这是一种归纳偏差。如果这种假设对某组数据来说并不是最优的解释方法，那么就会导致模型不稳定甚至重建失败。而 $\pi^{3}$ 采用了一个排列无关的架构对 affine-invariant 的相机位姿以及 scale-invariant 的 local点云进行预测，无需任何参考帧。这使得我们的模型对输入顺序有固然的鲁棒性，并且高度 scalable。这些优势使得我们的简单且无偏差的方法可以在各种任务上实现最先进的性能。包括相机姿态估计，单目/视频深度估计和密集点图重建。

Introduction

“Learning always beats inductive biases.” — Jonathan T. Barron, Radiance Fields and the Future of Generative Media

“学习永远比归纳偏差强。”

  • 近期，前馈神经网络在三维重建领域有了十分亮眼的进展，传统方法一般采用 BA 。代表的是 DUSt3R。

  • 但无论是以往的方法还是新方法，有一个关键的局限性一直存在：对选择一个单一、固定的参考视角的依赖。整个相机的坐标系统都建立在将这个关键帧作为参考的基础上。而这是一个从 SfM 与 MVS 中继承过来的做法。

    • 我们认为，这种做法引入了不必要的归纳偏差，从根本上限制了前馈网络的性能和鲁棒性。
    • 我们通过实验证明，这种对任意的参考帧的依赖让现有的方法（包括 VGGT）对初始的视角选择高度敏感。如果初始选择较差，重建质量将受到极大影响，这也阻碍着我们实现一个鲁棒和 scalable 的系统。

  • 为了克服这种局限性，我们引入了 $\pi^{3}$，一种 鲁棒，scalable，排列无关的方法，其可以消除视觉几何学习任务中基于参考视图的偏差。

    • $\pi^{3}$ 接受多样的输入：单一图像，视频序列，或者无序图像集（无论是静态场景还是动态场景），且不需要指定参考视图。
    • 我们的模型会对每一帧预测其 affine-invariant 的相机位姿以及 scale-invariant 的 local点云，而基于的都是每帧图像自己本身的相机坐标系。
    • 通过在设计中避开依赖图像顺序的组件（比如帧 index 位置编码），并采用一个在 view-wise 和全局注意力之前交替的 transformer 架构，$\pi^{3}$ 实现了真正的排列无关。
    • 这保证了视觉输入和重建的几何之间的一致的一对一映射，使模型在输入顺序上有固有的鲁棒性，并不受参考视图选择问题的影响。

  • 这样的设计有显著的优势

    • 首先，其非常 scalable。
    • 其次，收敛速度很快。
    • 最后，更鲁棒。
    • 与之前的方法比，当参考帧变化的时候，我们的方法性能下降最小，标准差也很低。

Related Work

传统三维重建

这些方法利用了多视角几何的特性建立图像之间的特征相关性表征，从中估计相机位姿，并生成稠密点云。虽然鲁棒，尤其是在受控环境下；但这些方法需要复杂，多阶段的流程。
并且 BA 方法的时间成本很高。

前馈式三维重建

单次前馈生成 3D 结构。
比如 Dust3R，成对地处理图像并按照第一个相机位姿的坐标系来预测点云。这种方法对于两个视角很有效，但是 scale 到大场景的话需要做全局的对齐，这个对齐就可能又耗时又不稳定。
后来的一些工作尝试克服这个局限性，Fast3R 通过同时对数千张照片进行推理，从而消除了对全局对齐阶段的要求。
还有一些方法探索将学习问题简化，比如，FLARE 将任务拆分成几段，首先估计对每个图片的相机位姿进行预估，然后将每个相机坐标系下的点云进行预估，最后进行合并。现在的 SOTA 方法 VGGT，利用了多任务学习和大规模数据集得到了很好的准确性和模型表现。
这些方法的统一特征是他们依赖于将预测的 3D 结构锚定在指定的参考系中。我们的工作提出了一种完全不同的方法。

Method

PI 结构

为了保证模型的输出是和输入视角的顺序无关的，我们让我们的网络 $\phi$ 具有排列无关性。

输入 $N$ 张图片，成为一个集合 $S=(I_{1},\dots,I_{N})$, 每张图片的尺寸 $I_{i} \in \mathbb{R}^{H \times W \times 3}$, 网络 $\phi$ 将这个序列映射到输出元组上：

$$\phi(S)=((T_{1},\dots,T_{N}),(X_{1},\dots,X_{N}),(C_{1},\dots,C_{N}))$$
这里，$T_{i} \in SE(3) \subset \mathbb{R}^{4 \times 4}$ 是相机位姿，$X_{i} \in \mathbb{R}^{H \times W \times 3}$ 是对应的与像素对齐的 3D 点云的空间坐标，在该图片对应相机坐标系中的表示。 $C_{i} \in \mathbb{R}^{H \times W}$ 是 $X_{i}$ 的置信度矩阵。

对于一种排列 $\pi$，记 $P_{\pi}$ 为排列序列顺序的运算符（可以理解为定义了一个顺序 $\pi$, 然后 $P_{\pi}$ 就是按照这个顺序打乱的操作），网络满足：
$$\phi(P_{\pi}(S))=P_{\pi}(\phi(S))$$
那么我们可以发现，如果对初始的 S 的顺序按 $\pi$ 进行打乱，那么通过模型 $\phi$ 得到的结果，也会是按照 $\pi$ 进行打乱的。所以这种特性也保证了我们的每个图片输入和对应的输出是一对一映射的关系。 而这就带来了一些优势。

  • 首先，重建的质量就与参考视角的选择无关了。
  • 其次，模型对不确定性或者噪声更加鲁棒了。
  • 最后，这种架构有很强的 scalability。


传统的方法都需要在 ViT 的 token 上加上参考视角的 token 或者直接与一个 learnable 的 reference embedding 进行相加（比如 VGGT 中的 camera token），我们的方法由于保证了 PI 特性，所以不需要这些步骤。使用相对监督，就可以自然地对输入视角的顺序鲁棒。


这是 VGGT 的 pipeline 图

为什么 VGGT 里面是这个架构？
VGGT 中，首先在进行 transformer 架构之前加入一个随机初始化的 camera token，然后整体的架构就是一个交替的 global attention 和 frame attention 的结构。Global attention 的逻辑很简单，就是为了确保注意力具有全局一致性，而 frame attention 就是 VGGT 的核心亮点之一。

如果我们直接将输入的图片们直接输入进 transformer 中，那么对于 appearance 非常相似的两个图像，（作者举的例子）模型很有可能将这两个图像看做是一个图像来的。那么一个非常自然的想法就是我们先给图像加上一个 frame index 位置编码，但是这会导致我们推理的时候，可以输入的序列的长度会受到训练集中最长的序列长度的影响，输入长度不灵活。那么这里在经过一次全局注意力的处理之后，我们将来自同一个 frame 的 tokens 自己内部做一个 Self-Attention，那么模型就可以自然地区分开哪些 tokens 来自哪些帧。

我们的 pipeline 从基于 DINOv2 （和 VGGT 一样的）的 backbone把每个视角 embedding 成一个 patch token 的序列开始，然后这些 token 将会经过一些交替的 view-wise层和全局自注意力层（与 VGGT 一样的），最后经过一个 decoder 就可以得到输出了。

Scale-invariant Local Geometry

对于每个输入图像 $I_{i}$，模型预测一个 pixel-aligned 三维点云 $\hat{X}{i}$ ，每个点云都是基于自己的相机坐标系，所以也叫 local geometry。
单目深度估计中固有的问题是 scale ambiguity。为了解决这个问题，我们模型预测点云的过程中，会基于给定场景的 $N$ 张图片学习一个位置且一致的 scale factor。
那么，训练的阶段就需要将所有帧预测出来的点云与其对应的 GT 做对齐。也就是对齐 $(\hat{X}{1},\dots,\hat{X}{N})$ 和 $(X{1},\dots,X_{N})$ 。这个对齐的操作可以通过求解一个最佳的 scale factor $\hat{s}$ 来实现。$\hat{s}$ 最小化了整个图像序列中 depth-weighted的 L1 距离：
$$s^{*}=\underset{x}{argmin}\sum_{i=1}^{N}\sum_{j=1}^{H \times W} \frac{1}{z_{i,j}}\lvert\lvert s \hat{x}{i,j}-x{i,j} \rvert\rvert {1}$$
其中 $\hat{ x}{i,j}$ 是预测出来的第 j 个像素点的三维点云坐标。$x_{i,j}$ 是对应的 GT，$z_{i,j}$ 是对应的 GT 的 depth 值，同时也是 $x_{i,j}$ 的 $z$ 轴坐标（因为平面是和相机平面对齐的）。
整个解决的过程使用一个 ROE solver。

最后可以定义出来 loss：
$$\mathcal{L}{points}=\sum{i=1}^{N}\sum_{j=1}^{H \times W} \frac{1}{z_{i,j}}\lvert\lvert \hat{s} \hat{x}{i,j}-x{i,j} \rvert\rvert {1}$$
为了鼓励对局部平滑表面的重建，我们引入了法向 loss $\mathcal{L}{normal}$ ，对每个 $\hat{X}{i}$ 中的点，对应的法向量 $\hat{n}{i,j}$ 通过到在图像网格上的邻居点的向量进行叉积计算得到。然后我们只需要最小化 GT 中的法向量与我们估计的法向量的角度差就行了。
$$\mathcal{L}{normal}=\sum{i=1}^{N}\sum_{j=1}^{H \times W}arccos(\hat{n}{i,j}\cdot n{i,j})$$
置信度矩阵 $C_{i}$ 通过一个二元交叉熵损失计算，每个点标记为 1 还是 0 取决于其 $\frac{1}{z_{i,j}}\lvert \lvert \hat{s}\hat{x}{i,j}-x{i,j} \rvert \rvert_{1}$ 项的值有没有小于一个阈值 $\epsilon$。

Affine-invariant Camera Pose

模型的排列不变性与多视角重建的 scale ambiguity 共同决定了我们输出的相机位姿 $(\hat{T}{1},\dots,\hat{T}{N})$ 与真值之间至少差了一个相似变换 *（我的理解是一个由于只能确定相机之间相对位姿所引起的旋转加平移的差距，以及一个由于 scale ambiguity 引起的 scale 上的差距 （注意这里的 scale factor 是全局的），我们在设计评价标准的时候应该将这一点考虑在内）*
为了解决全局参考帧的 ambiguity 问题，我们使用视角之间的相对位姿来进行监督（好像也没有更好的方法）。定义预测的从 j 到 i 视角的相对位姿为 $\hat{T }{i \leftarrow j}$：
$$\hat{T}{i \leftarrow j}=\hat{T}{i}^{-1}\hat{T}{j}$$
每一个 $\hat{T}{i \leftarrow j}$ 都是由一个旋转 $\hat{R}{i \leftarrow j}$ 和一个平移 $\hat{t}{i \leftarrow j }$ 组成的。旋转本身不受到全局变换的影响，但是平移的具体量会受到 global scale factor 的影响。所以我们利用一下上一节中估计的 $\hat{s}$，我们就可以对平移的值有一个矫正了。
下面定义 camera loss。Camera loss 由旋转 loss 和平移 loss 加权得到，然后对所有的图像对进行平均:
$$\mathcal{L}{cam}=\frac{1}{N(N-1)}\sum_{i\neq j}(\mathcal{L}{rot}(i,j),+\lambda\mathcal{L}{trans}(i,j))$$
旋转 loss 只需要最小化预测的相对旋转和真值之间的差距:
$$\mathcal{L}{rot}(i,j)=\arccos(\frac{Tr((R{i \leftarrow j} )^{T}\hat{R}{i \leftarrow j})-1}{2})$$
这里详细解释一下为什么要这么写：
首先 $Tr((R{i \leftarrow j} )^{T}\hat{R}{i \leftarrow j})$ 是Frobenius product，写出来其实就是逐元素求积再相加。
由 Rodrigues 旋转公式，我们可以得到 $R=I+\sin(\theta)[v]{\times}+(1-\cos(\theta)[v]{\times})$, 其中 $[v]{\times}$ 是单位旋转轴向量的反对称矩阵。
（但是当我推完这 Rodrigues 之后发现这好像用不太上）
我们首先想要知道的是，既然 $R$ 和 $\hat{R}$ 都属于 $SO (3)$，那么很自然地我们猜测 $(R_{i \leftarrow j} )^{T}\hat{R}_{i \leftarrow j}$ 也是属于 $SO (3)$ 的。
#### 证明 $R^T R’$ 是正交矩阵

首先，回顾一下正交矩阵的定义：一个矩阵 AA 是正交矩阵，如果满足：
$$A^T A = I$$
其中 $I$ 是单位矩阵。
考虑矩阵 $R^T R’$，其中 $R$ 和 $R’$ 都是属于 $SO(3)$ 的旋转矩阵，意味着它们是正交矩阵，因此：

$$R^T R = I \quad \text{和} \quad R’^T R’ = I$$

现在我们来检查 $R^T R’$ 是否满足正交矩阵的条件。我们计算其转置：
$$(R^T R’)^T = R’^T R$$

然后，计算其乘积：

$$(R^T R’)^T (R^T R’) = R’^T R R^T R’ = R’^T I R’ = R’^T R’ = I$$

这表明 $R^T R’$ 是正交矩阵，因为它满足$(R^T R’)^T (R^T R’) = I$。

证明 $\det(R^T R’) = 1$

接下来，我们检查 $det(R^T R’)$ 是否等于 1。根据行列式的性质，我们有：
$$\det(R^T R’) = \det(R^T) \det(R’)$$
由于 $R$ 和 $R’$ 都是旋转矩阵，且旋转矩阵的行列式为 1，即：
$$\det(R) = \det(R’) = 1$$
所以：
$$\det(R^T R’) = \det(R^T) \det(R’) = 1 \times 1 = 1$$

因此，$\det(R^T R’) = 1$，所以我们就验证了 $R^{T}R’$ 矩阵也是属于 $SO (3)$ 的，那么我们观察 $Tr((R_{i \leftarrow j})^{T}\hat{R}{i \leftarrow j})$ 的范围，其实就是看 $Tr (R)$ 的范围。
我们知道对于旋转矩阵 $R$，其特征值为 $1,e^{i\theta},e^{-i\theta}$。那么其迹就是特征值之和，即 $1+e^{i\theta}+e^{-i\theta}=1+2\cos(\theta)$，那么也就不难解释整个式子为什么设计为：
$$\mathcal{L}{rot}(i,j)=\arccos(\frac{Tr((R_{i \leftarrow j} )^{T}\hat{R}_{i \leftarrow j})-1}{2})$$

对于平移 loss，我们将尺度变换过后的预测值与真值的相对平移进行比较，使用 Huberloss 来计算，因为其对于离群值非常鲁棒：
$$\mathcal{L}{trans}(i,j)=\mathcal{H}{\delta}(s^{*}\hat{t}{i\leftarrow j}-t{i\leftarrow j})$$

我们的 affine-invariant 相机模型建立在一个关键点之上：现实世界中，相机的路径是高度结构化的，而不是随机的。（也许这里也会是引入了一个归纳偏差的？） 它们通常位于低维流形上，例如，围绕着物体运行的相机沿着球体移动，而车载相机沿着曲线移动。我们在图 5 中分析了预测的位姿分布的结构，发现我们预测的相机位姿的方差集中在比 VGGT 少得多的主成分上，说明我们的输出是一个低维结构。


从这个图中我们可以看出，我们的模型对于位置来说，在前两个主成分上的特征值明显大于第三个主成分上的特征值，而 VGGT 的则都不明显，说明我们的模型在 position 的输出上具有显著的低维性。同理，rotation 也是一样。


我们可以看到我们的模型预测的相机位姿，明显有更好的低维特性。

Ablation Study

为了进行公平的比较，我们使用 VGGT 的修改版本，禁用了用于 tracking 和 global map heads，作为 baseline（模型 1）。我们依次集成了 scale-invariant 点云和 affine-invariant 相机位姿，分别产生了模型 2,3。模型 1 类似于 VGGT，模型 1,2 都使用附加标记来指定参考系，并且预测坐标系内的相机姿势，而模型 3 是完全排列等效的。所有模型都对 224*224 的图像进行了 80 个 epoch 的训练。所有模型都是从头开始训练的，但编码器除外，是用 DINOv2 权重初始化的。添加 scale invariant 将指标从 0.1498 提高到 0.1145，完整模型在加上了 affine-invariant 的相机位姿之后，将指标进一步提高到了 0.0625。这证明我们的两个组件都是必不可少的，并且对我们的架构最终的准确性和稳健性做出了巨大贡献。关于模型 2,3 的细节在附录中。

Conclusion

在这项工作中，我们引入了π3，这是一种前馈神经网络，它通过消除对固定参考视图的依赖，为视觉几何重建提供了一种新的范式。通过利用完全排列等变架构，我们的模型本质上对输入排序具有鲁棒性，并且具有高度可扩展性。这种设计选择消除了以前方法中发现的关键感应偏差，使我们简单而强大的方法能够在各种任务上实现最先进的性能，包括相机姿势估计、深度估计和密集重建。π3 表明无参考系统不仅可行，而且可以产生更稳定、更通用的 3D 视觉。

Appendix

Architecture Details

编码器和交替注意力模块和 VGGT 中的相同，不同之处在于交替注意力层模块仅使用 36 层，而 VGGT 使用 48 层。相机位姿，点云和置信度矩阵的解码器共享相同的结构，但是不共享权重。该架构是一个轻量级的 5 层 transformer，专门将自注意力应用于每个单独图像的特征。在解码器之后，输出头因任务而异。点云图和置信度矩阵的 head 由一个简单的 MLP 和一个 pixel shuffle 的操作。对于相机位姿，其 head 来自于 Reloc3r，并且使用 MLP、Average Pooling，和另一个 MLP。旋转一开始使用 9D 的形式预测，然后通过 SVD 正交化转换为 3*3 的旋转矩阵。

Training Details

我们分两个阶段训练，首先在 224*224 的低分辨率图片上进行训练，然后对总像素数在 100000，以及 255000 的随机分辨率图像进行微调。使用 VGGT 类似的动态 batch 大小调整策略，第一阶段，每个 GPU 进行 64 张图像的采样，第二阶段每个 GPU 进行 48 图像的采样。模型不是从头开始训练的，是从预训练的 VGGT 中初始化编码器和交替注意力模块的权重，并在训练期间保持编码器冻结。对于每个阶段使用 adams 优化器，学习率从 1e-4 开始衰减。在 16 个 A100 上训练第一阶段，在 64 张 A100 上训练第二阶段。

Limitations

  • 无法处理透明物体，因为模型没有明确考虑复杂的光传输现象。
  • 我们重建的集合形状缺乏相同水平的细粒度细节（与基于 Diffusion 的方法比）。
  • 点云生成依赖于使用具有 pixel shuffle 的 MLP 的简单上采样机制。虽然高效，但是这种设计会引入明显的网格状伪影，特别是在重建不确定性高的区域。

Description

A config array is a way of managing configurations that are based on glob pattern matching of filenames. Each config array contains the information needed to determine the correct configuration for any file based on the filename.

Background

In 2019, I submitted an ESLint RFC proposing a new way of configuring ESLint. The goal was to streamline what had become an increasingly complicated configuration process. Over several iterations, this proposal was eventually born.

The basic idea is that all configuration, including overrides, can be represented by a single array where each item in the array is a config object. Config objects appearing later in the array override config objects appearing earlier in the array. You can calculate a config for a given file by traversing all config objects in the array to find the ones that match the filename. Matching is done by specifying glob patterns in files and ignores properties on each config object. Here’s an example:

When you retrieve a configuration for foo.json, only the first config object matches so handler is equal to jsonHandler; when you retrieve a configuration for package.json, handler is equal to packageJsonHandler (because both config objects match, the second one wins).

  • The config array caches configs, so subsequent calls to getConfig() with the same filename will return in a fast lookup rather than another calculation.
  • A config will only be generated if the filename matches an entry in a files key. Any config with a files key entry ending with /** or /* will only be applied if another entry in the same files key matches or another config matches.

Determining Ignored Paths

You can determine if a file is ignored by using the isFileIgnored() method and passing in the absolute path of any file, as in this example:

A file is considered ignored if any of the following is true:

  • It’s parent directory is ignored. For example, if foo is in ignores, then foo/a.js is considered ignored.
  • It has an ancestor directory that is ignored. For example, if foo is in ignores, then foo/baz/a.js is considered ignored.
  • It matches an ignored file pattern. For example, if **/a.js is in ignores, then foo/a.js and foo/baz/a.js are considered ignored.
  • If it matches an entry in files and also in ignores. For example, if **/*.js is in files and **/a.js is in ignores, then foo/a.js and foo/baz/a.js are considered ignored.
  • The file is outside the basePath. If the basePath is /usr/me, then /foo/a.js is considered ignored.

For directories, use the isDirectoryIgnored() method and pass in the absolute path of any directory, as in this example:

A directory is considered ignored if any of the following is true:

  • It’s parent directory is ignored. For example, if foo is in ignores, then foo/baz is considered ignored.
  • It has an ancestor directory that is ignored. For example, if foo is in ignores, then foo/bar/baz/a.js is considered ignored.
  • It matches and ignored file pattern. For example, if **/a.js is in ignores, then foo/a.js and foo/baz/a.js are considered ignored.
  • If it matches an entry in files and also in ignores. For example, if **/*.js is in files and **/a.js is in ignores, then foo/a.js and foo/baz/a.js are considered ignored.
  • The file is outside the basePath. If the basePath is /usr/me, then /foo/a.js is considered ignored.

Important: A pattern such as foo/** means that foo and foo/ are not ignored whereas foo/bar is ignored. If you want to ignore foo and all of its subdirectories, use the pattern foo or foo/ in ignores.

Caching Mechanisms

Each ConfigArray aggressively caches configuration objects to avoid unnecessary work. This caching occurs in two ways:

  1. File-based Caching. For each filename that is passed into a method, the resulting config is cached against that filename so you’re always guaranteed to get the same object returned from getConfig() whenever you pass the same filename in.
  2. Index-based Caching. Whenever a config is calculated, the config elements that were used to create the config are also cached. So if a given filename matches elements 1, 5, and 7, the resulting config is cached with a key of 1,5,7. That way, if another file is passed that matches the same config elements, the result is already known and doesn’t have to be recalculated. That means two files that match all the same elements will return the same config from getConfig().
